## Context
BDOs generate fillable PDF forms for borrowers to complete. These forms need to be accessible to borrowers via a shareable link without requiring the borrower to create an account or log in. The link must be secure enough to prevent unauthorized access while being easy to share via email.

## Goals / Non-Goals

### Goals
- Provide a frictionless document download/upload experience for borrowers
- Generate secure, unguessable tokens for form access
- Track form downloads and uploads for BDO visibility
- Implement one-way upload (borrowers cannot retrieve uploaded files)
- Keep the implementation simple with existing tech stack (Next.js, Firestore, Firebase Storage)

### Non-Goals
- Full borrower authentication/account management
- Two-way document editing or collaboration
- Email notifications (can be added later)
- Document signing integration (separate future feature)
- Multiple upload sessions (token is single-use concept)

## Decisions

### Decision 1: URL Structure
Use `/forms/[token]` route with cryptographically secure tokens.

**Rationale**: Using a token instead of projectId prevents enumeration attacks and provides a layer of security. The `/forms/` path is clear and distinct from the existing `/borrower/` authenticated routes.

**Alternatives considered**:
- `/borrower-portal/[token]` - Confusing with existing disabled route
- `/download/[token]` - Too generic, doesn't convey full functionality
- `/p/[token]` - Too short, not descriptive

### Decision 2: Token Generation
Generate 32-character URL-safe random tokens using `crypto.randomBytes(24).toString('base64url')`.

**Rationale**: Base64url encoding provides compact, URL-safe tokens. 24 bytes (192 bits) of entropy is sufficient for security while keeping URLs manageable.

**Token storage**: Store in Firestore document at `projects/{projectId}/formPortal` with:
```typescript
{
  token: string,           // The magic link token
  createdAt: Timestamp,
  expiresAt: Timestamp,    // Optional: token expiration (e.g., 30 days)
  projectId: string,       // Reference back to project
  generatedFormsIds: string[], // List of form IDs available for download
}
```

### Decision 3: File Upload Storage
Store uploaded files in Firebase Storage under `borrower-uploads/{projectId}/{timestamp}_{filename}`.

**Rationale**: Keeps uploads separate from other project files. Timestamp prefix prevents filename collisions.

**Metadata storage**: Store upload records in Firestore at `projects/{projectId}/borrowerUploads/{uploadId}`:
```typescript
{
  filename: string,
  originalName: string,
  storagePath: string,
  uploadedAt: Timestamp,
  fileSize: number,
  mimeType: string,
  relatedFormId?: string,  // Optional link to which form this fulfills
}
```

### Decision 4: One-Way Upload Implementation
The public portal page only shows download links for generated forms. Uploaded files are stored but not displayed or retrievable from the public portal.

**Rationale**: Simplifies UI, prevents confusion, and adds a layer of security. BDOs can view/download uploaded files from their admin interface.

### Decision 5: Download Tracking
When a borrower downloads a form, update the form's `downloadedAt` timestamp in Firestore.

**Rationale**: Allows BDOs to see which forms have been accessed by the borrower.

## Risks / Trade-offs

### Risk: Token Leakage
If a token URL is shared or leaked, anyone with the link can access the forms.

**Mitigation**:
- Tokens can be regenerated by BDO (generates new token, invalidates old)
- Optional token expiration (30 days default)
- Logged access for audit trail

### Risk: Large File Uploads
Borrowers might upload very large files.

**Mitigation**:
- Set file size limit (e.g., 25MB per file)
- Accept only PDF, images, and common document formats
- Show clear error messages for rejected files

### Risk: Spam/Abuse
Public endpoint could be abused.

**Mitigation**:
- Token validation required before any upload
- Rate limiting on upload endpoint
- File type validation

## Data Model Changes

### New Collection: `projects/{projectId}/formPortal`
Single document storing portal token and metadata.

### New Subcollection: `projects/{projectId}/borrowerUploads`
Documents for each uploaded file.

### Modified: `projects/{projectId}/generatedForms/{formId}`
Add `downloadedAt` timestamp field.

## Migration Plan
No migration needed - this is a new capability. Existing projects will get tokens when BDOs regenerate forms or when the "Copy Portal Link" action generates a new token.

## Open Questions
- Should tokens expire? (Recommendation: yes, 30 days with option to regenerate)
- Should we send email notifications when borrower uploads? (Recommendation: defer to future change)
- Maximum number of files borrower can upload? (Recommendation: 10 files, 25MB each)
